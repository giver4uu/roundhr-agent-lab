# 보리 인터뷰: 지원자 개인정보 인증 기능 초기 논의

**날짜**: 2025-10-20
**인터뷰 대상**: 보리 (HR 실무 전문가)
**목적**: 지원서 작성 중 저장 및 제출 전 수정 기능을 위한 개인정보 인증 방식 검증

---

## 배경

- 지원자가 지원서 작성 중에 저장할 수 있고, 제출 전에 언제든지 수정 가능하도록 하는 기능 추가 검토
- 타 솔루션과 같은 회원가입 방식은 지양하고자 함
- 초기 제안: 소셜 로그인 (Google, LinkedIn, Kakao)

---

## 주요 질문 및 답변

### 1. 지원서 작성 중 저장 못 해서 생기는 문제

**실제 발생 상황:**
- **모바일 환경 이탈**: 지원자의 40-50%가 모바일로 지원서 작성
  - 출퇴근길, 점심시간에 작성 중 전화나 배터리 문제로 데이터 소실
  - 경력직은 자기소개서가 길어서 모바일로 한 번에 완료 어려움

- **직장인 지원자**: 회사에서 몰래 작성하다가 상사 출현 시 급히 창 닫기
  - 스타트업/중소기업 채용 시 이직 준비자들이 회사 컴퓨터에서 작업

- **신중한 지원자**: 자소서를 Word에 미리 작성하지만, 파일첨부나 선택형 질문이 많으면 사이트에서 직접 작성 필요 (30분~1시간 소요)

**실제 사례:**
> "작년에 개발자 채용할 때 정말 우수한 분이 지원을 3번이나 시도하다가 포기한 적이 있어요. 나중에 LinkedIn으로 직접 연락했더니 '매번 작성하다가 회의 들어가서 날아갔다'고 하시더라고요. 그 분 결국 경쟁사로 가셨어요."

### 2. 제출 전 수정 빈도와 내용

**주로 수정하는 내용:**
- **오탈자와 표현 다듬기**: 작성 후 재독 시 어색한 표현 발견
- **포트폴리오/첨부파일**: 잘못된 버전 업로드 후 교체
- **추가 경험**: 지원 후 생각난 프로젝트나 경험 추가

**핵심 인사이트:**
> "지원자들이 제출 버튼 누르기 전에 한 번 더 검토하고 싶어 한다는 거예요. 지금은 '제출하면 끝'이라는 압박감 때문에 완벽하게 준비될 때까지 아예 시작을 안 하거나, 너무 오래 붙들고 있다가 공고가 마감되는 경우가 많아요."

### 3. 인증 방식 제안 (소셜 로그인)

**보리의 추천:**

**소셜 로그인 (Google, LinkedIn, Kakao) 제공**
- 요즘 누구나 소셜 로그인 계정 보유
- 특히 LinkedIn은 이력서 정보 자동 입력 가능 (일석이조)
- "인증 방식"보다 "지원자가 나중에 다시 돌아올 수 있는 편리한 방법"이 핵심

**이메일/전화번호 인증의 문제점:**

**이메일 코드 방식:**
- 회사 이메일 퇴사 후 접근 불가
- 스팸함에 들어가면 찾기 어려움
- 매번 코드 받는 게 번거로움
- 중복 지원 방지 어려움

**휴대폰 인증:**
- 개인정보 수집 동의 절차 복잡
- SMS 인증 비용 발생
- 외국인 지원자 문제

### 4. HR 담당자 입장 우려사항

**전혀 문제없음. 오히려 선호:**
- ✅ **완성도 높은 지원서**: 여러 번 수정한 고품질 지원서 선호
- ✅ **진정성**: 시간 들여 수정한다는 것은 회사 관심도가 높다는 의미
- ✅ **정확한 정보**: 오탈자나 잘못된 파일 수정 가능

**주의사항:**
- **버전 관리**: "최종 제출 일시" 명확히 표시 필요
- **제출 후 수정 불가**: 제출 버튼 명확히 구분, 제출 후에는 수정 불가 원칙

### 5. 임시 저장 데이터 보관 기간

**추천: 채용 공고 마감일 + 7일**

**이유:**
- 공고 마감일까지 당연히 보관 (지원자가 언제든 완성 가능)
- 마감일 + 7일: 마감 놓친 지원자나 다른 포지션 재활용 고려

**제안 정책:**
```
- 임시 저장 후 30일간 미접속 시 자동 삭제 (단, 공고 마감 전 유지)
- 공고 마감 후 7일 뒤 자동 삭제
- 삭제 3일 전 이메일 알림
```

**법적 고려사항:**
- GDPR/개인정보보호법: 동의한 목적(채용) 종료 후 삭제 필요
- 시스템 부하: 미완성 지원서 수백 개 누적 시 DB 관리 어려움

### 6. 실제 효과와 우선순위

**효과에 대한 현실적 평가:**

**데이터 기반 인사이트:**
- 지원서 양식 간소화: 35% → 58% 완료율 (23%p 증가)
- 임시 저장 기능 추가: 58% → 62% 완료율 (4%p 증가)

**중요한 이유:**
1. **프리미엄 지원자 확보**: 우수한 지원자일수록 신중하게 준비
2. **브랜드 이미지**: "지원자 경험 중시" 이미지 (특히 개발자/디자이너)
3. **기본 기능**: 2025년에 임시 저장 없는 지원서는 시대착오적

**함께 고려할 기능들:**
1. **자동 저장**: 지원자가 저장 버튼 누르지 않아도 자동 저장 (더 중요)
2. **진행률 표시**: "60% 완료" 같은 표시가 완료율에 더 큰 영향
3. **모바일 최적화**: 임시 저장보다 모바일 작성 환경이 우선
4. **이력서 자동 파싱**: 기본 정보 수동 입력이 가장 귀찮음

---

## 우선순위 제안

### Phase 1 (Must Have)
- ✅ 소셜 로그인 (Google, LinkedIn)
- ✅ 자동 저장 (로컬스토리지)
- ✅ 모바일 최적화

### Phase 2 (Should Have)
- 계정 기반 임시 저장
- 이메일 알림 ("작성 중인 지원서")
- 진행률 표시

### Phase 3 (Nice to Have)
- 지원서 템플릿 재활용 (같은 회사 다른 포지션)
- 버전 히스토리

---

## 핵심 인사이트

1. **문제의 심각성**: 실제로 우수 인재가 저장 기능 없어서 이탈하는 사례 존재
2. **압박감 제거**: "제출하면 끝" 압박이 지원 시작 자체를 지연시킴
3. **편의성 우선**: "인증 방식"이 아닌 "편리하게 돌아올 수 있는 방법"이 핵심
4. **HR 선호**: 여러 번 수정한 완성도 높은 지원서를 선호
5. **전체 경험 개선**: 임시 저장만으로는 부족, 전체 지원자 경험(Candidate Experience) 개선의 일부로 접근 필요
