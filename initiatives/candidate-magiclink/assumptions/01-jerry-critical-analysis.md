# 제리 분석: 소셜 로그인 방식의 비판적 검토

**날짜**: 2025-10-20
**분석자**: 제리 (Senior PM)
**목적**: 보리가 제안한 소셜 로그인 방식의 잠재적 문제점 및 리스크 분석

---

## Executive Summary

보리의 인사이트는 실무 관점에서 가치가 있지만, 소셜 로그인을 Phase 1 Must Have로 진행하기 전에 **멈춰서 생각해봐야 할 심각한 문제점들**이 존재함.

**결론**: 소셜 로그인은 문제가 너무 많음. **Magic Link (이메일 인증 링크)** 방식을 대안으로 제안.

---

## 1. 비즈니스 관점의 문제점

### A. B2B 고객(기업 HR)의 우려사항

**브랜드 인식 문제**
- 지원자가 "회사 채용 페이지"가 아닌 "Google/LinkedIn 로그인 화면"을 먼저 봄
- 기업 브랜딩이 희석됨

**진입장벽 인식**
- HR이 "로그인 필요한 곳은 지원 안 한다"는 지원자 이탈 우려 제기 가능

**데이터 소유권 문제**
> "우리 회사 지원자 데이터가 왜 Google/LinkedIn 계정에 종속되어야 하나?"

**실제 시나리오 (대기업 HR):**
> "우리는 공정채용 원칙상 소셜 프로필 정보를 채용 프로세스에서 배제해야 합니다. LinkedIn 로그인은 프로필 사진, 현직장 정보 등이 자동으로 넘어올 수 있어서 문제가 될 수 있습니다."

### B. 지원자 데이터 관리 리스크

**데이터 무결성 문제:**
- 지원자가 소셜 계정 삭제 → 우리 시스템의 지원 데이터는?
- LinkedIn 계정 정보 변경 (이름 변경 등) → DB sync 이슈
- **중복 지원 문제**: 같은 사람이 Google로 한 번, LinkedIn으로 한 번, 이메일로 한 번 지원 → HR은 이를 어떻게 파악?

**데이터 품질 이슈:**
- 소셜 로그인으로 받은 이메일이 프록시 이메일이면? (예: noreply@linkedin.com)
- Kakao 계정은 이메일 없이도 가입 가능 → 지원자 연락 방법 없을 수 있음

---

## 2. 사용자 경험(UX) 관점의 맹점

### "모든 지원자에게 편리하다"는 가정의 함정

**세그먼트 1: 프라이버시 민감층**
- **재직자 이직 준비**: "취업 준비 중인 걸 현재 회사에 들키고 싶지 않아요"
- LinkedIn 로그인 시 "누가 내 프로필을 봤는지" 노출될 수 있음
- **재직자 이직 지원 시 큰 장벽**

**세그먼트 2: 소셜 계정 미보유층**
- LinkedIn: 한국에서 penetration 낮음 (특히 비IT 직군)
- Kakao: 20-30대는 높지만 40대+ 시니어층은?
- Google: 회사 이메일만 쓰는 사람들은 개인 Google 계정 없을 수도

**세그먼트 3: "소셜 로그인 불신층"**
- "왜 지원서 내는데 내 소셜 계정 정보가 필요한가?" 의심
- 특히 보안 의식 높은 개발자들

**실제 UX 시나리오:**
```
지원자: "LinkedIn으로 로그인했더니 프로필 사진이 자동으로 들어갔네?
        이거 blind recruitment 아니었나? 불공정한 거 아냐?"
```

### 보리가 놓친 점: 선택지의 역설

- Google + LinkedIn + Kakao + 이메일 가입 = 4개 선택지
- **선택 피로(Choice Paralysis)** 발생 가능
- 경쟁사가 "이메일만" 받으면 오히려 더 간단해 보일 수 있음

---

## 3. 기술적/운영적 리스크

### A. 플랫폼 의존성 리스크

**실제 발생 가능한 시나리오:**

1. **LinkedIn API 정책 변경**
   - 2024년 LinkedIn은 API 가격을 대폭 인상
   - Resume parsing API는 enterprise plan에서만 제공
   - "Phase 1에서 LinkedIn 로그인 붙였는데, 나중에 API 비용 감당 못해서 제거해야 할 수도"

2. **OAuth 서비스 다운타임**
   - 2023년 Kakao 로그인 장애 → 수많은 서비스 로그인 불가
   - 우리 서비스는 멀쩡한데 Kakao 때문에 지원자가 못 들어오면?

3. **계정 연동 해제 시나리오**
   ```
   지원자: Google 계정으로 가입 → 지원서 50% 작성 →
          (개인정보 우려로) Google 연동 해제 →
          다음날 다시 들어가려 하니 "계정을 찾을 수 없습니다"
   ```
   → 고객 지원팀 부담 증가

### B. 중복 계정 관리 악몽

**현실적으로 발생할 문제:**

```
같은 지원자 김철수:
- 1차: Google (personal.kim@gmail.com)으로 A 포지션 지원
- 2차: LinkedIn (chulsoo.kim@company.com)으로 B 포지션 지원
- 3차: 이메일 (kim.cs@naver.com)으로 A 포지션 재지원

HR 입장: "이 사람들이 같은 사람인지 어떻게 알죠?"
```

**해결책이 복잡함:**
- 이메일 매칭? → 각 플랫폼에서 다른 이메일 쓰면 불가
- 이름+생년월일? → 소셜 로그인으로는 생년월일 못 받음
- Phone number? → 소셜 로그인으로는 전화번호 기본 제공 안 됨

## 4. 법적/컴플라이언스 이슈

### A. 개인정보 수집 범위의 늪

**소셜 로그인 시 받아오는 정보:**
- Google: 이름, 이메일, 프로필 사진, (선택) 생년월일, 성별
- LinkedIn: 이름, 이메일, 프로필 사진, 현재 직장, 경력 정보
- Kakao: 이름(또는 닉네임), 이메일(선택), 프로필 사진, 성별, 연령대

**법적 리스크:**

1. **최소 수집 원칙 위반 가능성**
   - "지원서 임시 저장"이 목적인데 왜 프로필 사진이 필요한가?
   - 공정채용법: 사진, 연령, 성별 수집 제한 → 소셜 로그인은 자동으로 이걸 수집

2. **동의 절차의 복잡성**
   ```
   필요한 동의:
   1. 소셜 플랫폼 이용약관 동의
   2. 우리 서비스 개인정보 처리방침 동의
   3. 제3자 정보 제공 동의 (소셜 플랫폼 → 우리 서비스)
   4. HR 담당자에게 지원 정보 제공 동의
   ```
   → **UX 복잡도 증가**, 오히려 이메일 가입보다 번거로워질 수 있음

3. **GDPR "right to be forgotten" vs 채용 서류 보관 의무**
   - 지원자가 소셜 계정 삭제 → GDPR상 우리도 데이터 삭제해야 함
   - 하지만 채용 법규상 지원서는 일정 기간 보관 의무
   - **법적 충돌 가능성**

### B. 한국 특수성: 공정채용법

```
채용절차법 제4조의 10(채용서류 항목 제한)
- 용모, 키, 체중 등 신체 조건
- 사진 (업무 상 필요한 경우 제외)
```

**LinkedIn/Google 로그인 → 프로필 사진 자동 수집 → 법 위반 소지**

## 5. 대안: Magic Link (이메일 인증 링크)

### 작동 방식

1. 지원자가 이메일 입력
2. "저장 링크"를 이메일로 발송
3. 해당 링크로 언제든 돌아와서 수정 가능
4. **비밀번호 없음, 계정 생성 없음**

### 장점

- ✅ 소셜 계정 불필요
- ✅ 개인정보 최소 수집 (이메일만)
- ✅ UX 간단: 이메일 하나만 입력하면 끝
- ✅ 중복 지원 방지: 이메일로 자동 매칭
- ✅ GDPR/개인정보법 컴플라이언스 쉬움

### 단점

- ❌ 이메일 접근 불가 시 복구 어려움
- ❌ 스팸 메일함 걱정

### 벤치마크

- **Typeform, Notion, Slack** 모두 Magic Link 옵션 제공
- 특히 Typeform은 "Save & Continue Later" 기능을 Magic Link로 구현

---

## 6. 추천 접근법: Hybrid Approach

```
Phase 1:
- Default: Browser Local Storage + Magic Link (이메일)
- Optional: "LinkedIn으로 이력서 자동 입력" (지원자가 원하면 선택)

Phase 2 (데이터 확보 후):
- 지원자의 실제 선호도 분석
- 필요하다면 Google/Kakao 추가
```

### 이 접근의 장점

- ✅ MVP 빠르게 출시 (Magic Link는 구현 간단)
- ✅ LinkedIn Resume Parsing은 "부가 기능"으로 포지셔닝 (필수 아님)
- ✅ 실제 사용자 데이터로 Phase 2 결정 가능
- ✅ 법적/UX 리스크 최소화

## 보리에게 다시 물어봐야 할 질문들

1. **"LinkedIn Resume Parsing이 정말 critical한가요?"**
   - 지원자가 직접 이력서 업로드하는 게 더 정확하지 않을까요?
   - LinkedIn 프로필이 최신 상태가 아닌 경우도 많은데?

2. **"지원자 이탈의 진짜 원인이 뭔가요?"**
   - "저장 기능 없어서" 이탈하나요?
   - 아니면 "지원서가 너무 길어서" 이탈하나요?
   - 데이터 없이 추측으로 기능 만들고 있는 건 아닌가요?

3. **"보리가 실제로 관리하는 회사에서 소셜 로그인을 원할까요?"**
   - 보리의 회사 HR 정책상 소셜 프로필 정보 수집이 허용되나요?
   - 공정채용 원칙과 충돌하지 않나요?

## 최종 의견

### 문제 정의는 맞음
✅ "지원서 작성 중 저장 + 제출 전 수정" 은 좋은 기능

### 하지만 솔루션(소셜 로그인)은 의문
- ❌ **Over-engineering**: 문제 대비 솔루션이 너무 무거움
- ❌ **B2B 리스크**: 기업 고객의 컴플라이언스 우려
- ❌ **UX 가정 오류**: "모두가 소셜 로그인을 선호한다"는 증거 없음
- ❌ **기술 부채**: 3개 OAuth 관리 + 중복 계정 처리는 Phase 1에 과한 복잡도

### 제안

**Phase 1 Must Have:**
- Magic Link (이메일 인증 링크)로 "저장 & 나중에 계속하기" 구현
- Browser Local Storage로 즉시 자동저장

**Phase 1 Nice to Have:**
- LinkedIn "Resume Import" 버튼 (선택 사항)

**Phase 2 (검증 후):**
- 실제 지원자 행동 데이터 분석
- A/B 테스트: Magic Link vs Social Login
- 그 후 Google/Kakao 추가 여부 결정
