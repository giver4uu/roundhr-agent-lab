# 인터뷰 성공 기준

## Phase 1: 문제 정의 (Week 1-2)

### 인터뷰 수집 목표
- [ ] **Borry 심층 인터뷰 3-5회 완료**
- [ ] **다양한 페르소나 인터뷰 1-2회** (신입 채용 담당자, 채용 매니저 등)
- [ ] 각 인터뷰마다 스냅샷 생성 완료

### 데이터 품질 기준

#### 구체성 (Specificity)
- [ ] **최소 5개의 구체적 사례** 수집 (각 사례마다 날짜, 상황, 행동 포함)
- [ ] 각 사례에 **정량적 데이터** 포함 (시간, 빈도, 비용 등)
  - 예: "주당 5시간 소요", "면접 전 10분 정보 탐색", "월 20건 발생"

#### 감정과 맥락
- [ ] 각 스토리에 **감정 표현** 포함 (좌절, 불안, 답답함, 확신 부족 등)
- [ ] **5 Whys 추적** 최소 3개 사례 (표면적 불편 → 근본 원인)

#### 패턴 반복
- [ ] **최소 3개 인터뷰에서 반복되는 패턴** 식별
- [ ] 각 패턴의 **빈도와 심각도** 데이터 수집

### 주요 인사이트 카테고리

#### 1. 정보 탐색 실패 (최소 3개 사례)
- [ ] 의사결정 시점에 필요한 정보를 찾지 못한 사례
- [ ] 정보 탐색에 소요된 시간 정량화
- [ ] 찾지 못했을 때의 대응 방식 기록

#### 2. 맥락 손실 (최소 3개 사례)
- [ ] 과거 채용 과정의 맥락을 잃어버린 사례
- [ ] 팀원 간 지식 공유 실패 사례
- [ ] 채용 담당자 교체 시 지식 손실 사례

#### 3. 반복적 인지 부하 (최소 2개 사례)
- [ ] 동일한 작업을 반복하는 사례 (평가 기준 재작성, 인터뷰 질문 재생성 등)
- [ ] 과거 자료를 재사용하려다 실패한 사례

#### 4. 정보 단절 (최소 2개 사례)
- [ ] 채용 프로세스 단계 간 정보가 끊어지는 지점
- [ ] JD → 면접 평가 → 최종 결정 과정의 불일치 사례

#### 5. 암묵적 판단 기준 (최소 2개 사례)
- [ ] 이력서에 안 나오지만 중요한 평가 요소
- [ ] 신입 담당자가 놓치는 위험 신호
- [ ] 팀별로 다른 평가 기준

### Opportunity Statement 기준
- [ ] **최소 3개 Opportunity Statement 작성**
- [ ] 각 Opportunity마다 다음 포함:
  - [ ] **Problem Evidence** (3개 이상 인터뷰 출처)
  - [ ] **정량적 현재 상태** (예: "평균 10분 소요")
  - [ ] **명확한 개선 목표** (예: "5초 내 제공")
  - [ ] **측정 가능한 Success Metrics**

### 검증 기준
- [ ] **Borry 피드백:** "이 문제들이 실제로 우리가 겪는 핵심 pain point가 맞다"
- [ ] **비즈니스 임팩트 추정 가능:** 각 문제의 비용/시간 손실 정량화
- [ ] **솔루션 중립성 유지:** 인터뷰에서 온톨로지/LLM/AI 용어 사용하지 않음

## Phase 2: 솔루션 탐색 (Week 3-4)

### 전제 조건 (Gate)
**⚠️ Phase 1의 모든 성공 기준을 충족해야 Phase 2로 진행 가능**

### 솔루션 품질 기준
- [ ] **온톨로지 스키마 v0.1 작성**
  - [ ] 핵심 개념 10-15개 정의
  - [ ] 주요 관계 명시
  - [ ] **각 개념/관계가 해결하는 구체적 Opportunity 명시** (traceability)

- [ ] **AI 활용 시나리오 3-5개 작성**
  - [ ] 각 시나리오가 특정 Opportunity 해결
  - [ ] 온톨로지 + LLM 구체적 활용 방안 설명

- [ ] **Assumption Test Cards 5-10개 작성**
  - [ ] 리스키한 가정 식별
  - [ ] 각 가정의 테스트 방법 정의
  - [ ] 최소 3개 가정 프로토타입/컨셉 테스트로 검증 완료

### 검증 기준
- [ ] **온톨로지 개념이 인터뷰 사례로 근거 제시 가능**
- [ ] **기술팀 피드백:** "이 온톨로지로 구현 가능하다"
- [ ] **사용자 반응 긍정적** (프로토타입 테스트 5명 중 4명 "현재보다 낫다")

## Phase 3: 솔루션 검증 (Week 5-6)

### 전제 조건 (Gate)
**⚠️ Phase 2의 모든 성공 기준을 충족해야 Phase 3로 진행 가능**

### PRD 품질 기준
- [ ] **PRD 작성 완료** (`@guides/product/create-prd.mdc` 사용)
  - [ ] Problem Statement (Phase 1 인터뷰 근거)
  - [ ] Target Users (페르소나별 니즈)
  - [ ] Goals & Success Metrics (측정 가능)
  - [ ] Functional Requirements (온톨로지 스키마 기반)
  - [ ] Non-Goals (명확한 스코프)

- [ ] **엔지니어링 팀 리뷰 통과**
  - [ ] "이 스펙으로 개발 가능"
  - [ ] 기술적 리스크 식별 및 완화 방안 수립

### 프로토타입/POC 기준
- [ ] **프로토타입 테스트 완료** (실제 채용 담당자 5-10명)
  - [ ] 5명 중 4명 "현재보다 낫다" 평가
  - [ ] 주요 pain point 해결 확인

- [ ] **핵심 가정 80% 이상 검증 완료**

### Go/No-Go 의사결정 기준
- [ ] **비즈니스 케이스 완성**
  - [ ] 예상 ROI 추정
  - [ ] 개발 비용 산정
  - [ ] 시장 타이밍 분석
  - [ ] 경쟁 우위 명확화

## 실패 신호 (조기 경보)

### Phase 1 실패 신호
- ❌ 인터뷰에서 **추상적 의견만** 나옴 ("온톨로지 있으면 좋을 것 같아요")
- ❌ 패턴이 **반복되지 않음** (각 인터뷰마다 다른 문제 언급)
- ❌ 비즈니스 임팩트를 **정량화할 수 없음**
- ❌ Borry: "이건 우리 문제가 아닌 것 같은데요"

**대응:** 인터뷰 질문 방식 재검토, 더 구체적 사례 요청, 필요시 인터뷰 대상 확대

### Phase 2 실패 신호
- ❌ 온톨로지 개념이 **인터뷰 인사이트와 연결 안 됨**
- ❌ 솔루션이 **문제보다 복잡함** (간단한 필터로 해결 가능한데 온톨로지 도입)
- ❌ 가정 테스트 결과 사용자가 **제안 솔루션에 무관심**

**대응:** Phase 1로 돌아가 문제 재정의, 솔루션 단순화, 가정 재검증

### Phase 3 실패 신호
- ❌ PRD 리뷰: **"이게 왜 필요한지 모르겠다"**
- ❌ 프로토타입 테스트에서 사용자가 **기존 방식 선호**
- ❌ 개발 복잡도가 **예상 가치 초과**

**대응:** PRD 재작성, 프로토타입 재설계, 스코프 축소 또는 프로젝트 중단 검토

## 체크리스트 활용 방법

### 각 인터뷰 후
1. 이 문서의 해당 체크박스 체크
2. 부족한 부분 확인
3. 다음 인터뷰에서 보완할 항목 리스트 작성

### Week 2 종료 시점
1. Phase 1 모든 체크박스 확인
2. **Gate 통과 여부 판단**
3. 통과 시 Phase 2로 진행, 미통과 시 추가 인터뷰 계획

### 각 Phase 종료 시점
1. 해당 Phase 성공 기준 모두 충족 확인
2. 실패 신호 없는지 점검
3. 다음 Phase로 진행 또는 현재 Phase 반복 결정

---

**핵심 원칙:**
- **정량적 데이터 없으면 다음 단계로 진행 금지**
- **패턴 반복 확인 없으면 일반화 금지**
- **문제 검증 없으면 솔루션 탐색 금지**
